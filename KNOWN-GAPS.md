# Known Gaps

This library follows a deliberately narrow scope and an honest-documentation
discipline. What is *not* yet guaranteed is listed here so you can make an
informed decision before depending on it.

## Test-vector validation

**Status: VALIDATED against the official X-Wing Known-Answer-Test (KAT)
vectors** — all three vectors from the draft — with one honest residual limit
described below.

The vectors live in `tests/xwing_kat.json`. They were taken from the draft's
own `spec/test-vectors.json` (generated by the reference `xwing.py` over
ML-KEM-768 + X25519); the intermediate combiner components (`ss_m`, `ss_x`,
`ct_x`, `pk_x`) were derived **only** by running that reference, never from this
implementation, so the check is non-circular.

The harness (`tests/test_kat.py`) validates in three independent pieces, each
parametrized over all three vectors:

- **(A) Keygen KAT** — derives the X-Wing public key from the 32-byte seed via
  `liboqs-python`'s derandomized `generate_keypair_seed` (plus X25519) and
  asserts it equals the vector's `pk`, **byte-for-byte**. The SHAKE-256
  seed-expansion split is confirmed line-by-line against the reference
  `xwing.py`. Passes in default CI when `liboqs` is installed.
- **(C) Combiner KAT** — runs with *no backend*: feeds the reference-derived
  components through this package's `_combiner` and asserts it equals the
  vector's shared secret. This is what caught (and now guards against) the
  combiner byte-order: `XWingLabel` is appended **last**.
- **(B) Encaps KAT** — neither `liboqs` nor pyca/cryptography exposes *seeded*
  encapsulation, so this piece runs against the draft's reference encaps,
  enabled by pointing the `XWING_REF_ENCAPS` environment variable at the
  reference `xwing.py` directory. It confirms the reference reproduces the
  vector's `ct` and `ss`, and that our combiner agrees. Skips cleanly when the
  hook is unset.

**The one residual limit (honest):** the library's *own* `encapsulate()` uses
fresh randomness — neither backend offers a seeded encaps hook — so it is **not**
checked to emit the exact KAT ciphertext bytes directly. That path is covered
**transitively**: keygen-from-seed (A) and the combiner (C) are pinned to the
spec bytes, the reference encaps (B) reproduces the vectors, and the functional
round-trip tests confirm `encapsulate`/`decapsulate` agree. A direct,
byte-exact encaps KAT would require a seeded encapsulation API in one of the
backends.

## Constant-time / side-channel

- The lattice and curve operations run in **C** inside the chosen backend
  (pyca/cryptography or liboqs). Their constant-time properties are inherited
  from those implementations and are **not** re-verified here.
- The X-Wing combiner glue in this package is pure Python. It operates only on
  fixed-length values and performs **no secret-dependent branching**, but
  Python provides **no constant-time guarantee** (object allocation, GC, and
  interpreter behaviour are out of our control). Do not rely on this layer for
  timing-side-channel resistance in a hostile co-located environment.
- No formal side-channel analysis has been performed.

## Backend availability

- `cryptography` exposes ML-KEM only when its wheel was built against
  OpenSSL 3.5+, AWS-LC, or BoringSSL. The maintainers have stated some PQC
  APIs may be unavailable on plain-OpenSSL wheels. This package **probes** the
  backend at import and **falls back** to `liboqs-python` automatically; if
  neither works, `get_backend()` raises with install guidance.

## Secret-key encoding is backend-dependent

- The ML-KEM-768 secret key is serialized differently per backend
  (`cryptography` returns the 64-byte seed form; `liboqs` returns the
  ~2400-byte expanded form). An X-Wing secret key produced by one backend is
  therefore **not portable** to the other. Keys are usable only with the
  backend that generated them. A portable, self-describing key container is
  future work.

## Scope

- X-Wing only. No ML-KEM-1024 variant, no authenticated KEM, no signatures,
  no HPKE/TLS wiring. These are intentionally out of scope.
